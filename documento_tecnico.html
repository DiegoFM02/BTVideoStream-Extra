1. Descripción General

BTVideoStream es una aplicación Android nativa que permite a un dispositivo sin acceso a Internet (Cliente) buscar y reproducir videos de YouTube y TikTok a través de un segundo dispositivo con conexión (Servidor), usando Bluetooth clásico RFCOMM/SPP como único canal de comunicación. El desarrollo es íntegramente nativo en Kotlin con Jetpack Compose y arquitectura MVVM; no se utiliza ningún framework de terceros.

Responsabilidades del Servidor

  • Escuchar conexiones BT entrantes (rol pasivo)
  • Recibir SearchRequest y ejecutar búsqueda
  • Extraer la URL de stream del video solicitado
  • Descargar los bytes del video vía HTTPS
  • Servir desde caché si el video ya fue descargado
  • Enviar resultado en chunks ordenados por Bluetooth
  • Keepalive cada 2 s para mantener el enlace activo

Responsabilidades del Cliente

  • Iniciar conexión BT al servidor seleccionado
  • Enviar SearchRequest con fuente y consulta
  • Mostrar resultados recibidos en lista interactiva
  • Enviar VideoRequest con calidad elegida
  • Reensamblar chunks en archivo temporal MP4
  • Reproducir con ExoPlayer desde archivo local
  • Gestionar historial persistente (SharedPreferences)

3. Protocolo de Comunicación Bluetooth

3.1 Capa de transporte

Se utiliza Bluetooth Clásico SPP (Serial Port Profile) sobre sockets RFCOMM con un UUID de servicio fijo. El protocolo define una trama binaria de longitud variable con cabecera de 5 bytes:

┌──────────┬─────────────────────────────┬──────────────────────────────────┐ │ type │ length │ payload │ │ 1 byte │ 4 bytes (Big-Endian) │ N bytes │ └──────────┴─────────────────────────────┴──────────────────────────────────┘ Límite de seguridad: payload máximo = 20 MB (payloads mayores se rechazan).

3.2 Tipos de mensaje

SEARCH_REQUEST0x10UTF-8 "fuente|consulta"C → S
SEARCH_RESULTS0x11UTF-8 JSON array de resultadosS → C
VIDEO_REQUEST0x20[quality: 1 byte][videoId: UTF-8]C → S

3.3 Flujo de transferencia de video

Cliente Servidor │── VIDEO_REQUEST(videoId, quality) ──────────►│ │ │ descarga video (HTTPS) │◄─ STATUS("Descargando…") ───────────────────│ │◄─ VIDEO_CHUNK(index=0, 2048 bytes) ─────────│ │◄─ VIDEO_CHUNK(index=1, 2048 bytes) ─────────│ │◄─ … (N chunks de 2048 bytes) ───────│ │◄─ VIDEO_END ────────────────────────────────│ │ ensambla chunks → MP4 → ExoPlayer │

3.4 Reconexión automática

El cliente detecta desconexión mediante excepción en el socket de lectura. Ejecuta hasta 5 reintentos con espera progresiva (500 ms × intento). El servidor reanuda la escucha automáticamente tras perder un cliente. Ambos lados emiten notificaciones push (canal "Estado Bluetooth") al establecer o perder la conexión.

4. Decisiones de Diseño e Implementación

4.1 Extracción de stream sin APIs de pago

YouTube no ofrece una API pública gratuita para streams de video. La solución implementada consiste en obtener la página youtube.com/watch?v=ID con un User-Agent de navegador y parsear el objeto ytInitialPlayerResponse incrustado en el HTML. Este objeto contiene el array formats con URLs directas (muxed, video+audio combinados) que no requieren merge posterior. Se capturan las cookies de sesión del Set-Cookie de la respuesta y se reenvían en la descarga del CDN para autenticar la petición.

Se fuerza IPv4 en toda la pila de red (java.net.preferIPv4Stack=true) porque las URLs de CDN de YouTube incluyen un parámetro ip= firmado criptográficamente. Las extensiones de privacidad IPv6 de Android rotan la dirección temporal entre peticiones, causando un error 403 si la dirección de descarga difiere de la que generó el token. Al fijar IPv4, ambas peticiones (página y CDN) usan la misma dirección estable.

Se implementaron dos fallbacks automáticos: Cobalt API y tres instancias de Invidious con el parámetro local=true, que hace que el servidor de Invidious proxee el video a través de su propio nodo, eludiendo el IP binding del CDN.

4.2 TikTok — bypass de cadena SSL incompleta

El servicio TikWM (API no oficial de TikTok) presenta una cadena de certificados incompleta que Android rechaza por defecto. Se implementó un X509TrustManager permisivo y un HostnameVerifier que acepta cualquier host, aplicado únicamente a las conexiones hacia dominios de TikWM. La búsqueda se realiza con método POST application/x-www-form-urlencoded al endpoint /api/feed/search.

4.3 Transferencia binaria y ensamblado

El video completo se divide en fragmentos de 2048 bytes. Cada VIDEO_CHUNK lleva un índice de 4 bytes que permite reordenar fragmentos recibidos fuera de orden (chunkBuffer: Map<Int, ByteArray>). Al recibir VIDEO_END, el cliente concatena todos los chunks en orden, escribe el resultado en cacheDir/received_video.mp4 y lo entrega a ExoPlayer como FileMediaSource. El servidor aplica el mismo esquema de caché: si el video ya fue descargado en sesiones anteriores, se sirve directamente desde disco sin volver a contactar YouTube/TikTok.

4.4 Arquitectura MVVM y concurrencia

Cada rol (Servidor / Cliente) tiene su propio ViewModel con un SupervisorJob + Dispatchers.IO. El BluetoothManager expone el estado de conexión como StateFlow<ConnectionState> y los mensajes entrantes como SharedFlow<Message>, ambos consumidos desde los ViewModels mediante collectLatest. Las escrituras al socket pasan por un Channel<Message> de capacidad ilimitada que un único consumidor drena en orden FIFO, evitando condiciones de carrera al escribir desde múltiples corrutinas.

4.6 Temas y modo oscuro

La aplicación implementa dos esquemas de color Material 3: Guinda (color representativo del IPN, primario #7A1B3D) y Azul (color representativo de la ESCOM, primario #143C8C). Cada tema define paletas independientes para modo claro y modo oscuro, respondiendo automáticamente al sistema operativo vía isSystemInDarkTheme(). Un control ThemeController persiste la elección del usuario en SharedPreferences y expone el estado como MutableState observable por toda la jerarquía Compose.

5. Limitaciones Conocidas

LimitaciónCausa
YouTube HTTP 403 persistente El CDN de YouTube firma criptográficamente la URL con la IP del cliente y el token de sesión. Sin un motor JavaScript para generar el poToken requerido desde 2024, la descarga directa puede ser rechazada independientemente de las cabeceras enviadas.
Solo formatos muxed de YouTube Los streams adaptativos (DASH) separan video y audio en pistas distintas. Combinarlos requeriría ejecutar FFmpeg, que no está disponible como SDK nativo en Android sin librerías de terceros.
Velocidad de transferencia Bluetooth RFCOMM sobre Bluetooth Clásico ofrece ~2–3 Mbps teóricos. Videos de varios minutos en 360p (~30–60 MB) pueden tardar 20–60 segundos en transferirse.
Un cliente simultáneo El servidor acepta una única conexión RFCOMM activa. Múltiples clientes simultáneos requerirían un hilo de escucha por socket, fuera del alcance de este proyecto.
